# ISO/IEC 27001 Annex A 8.24 — Evidence-Chain Design

**Status: DESIGN ONLY.** This document designs a future *signed, timestamped
readiness report* that `qproof-tools` could emit as **evidence** supporting an
organization's ISO/IEC 27001:2022 Annex A **8.24 "Use of cryptography"** control.
It is the design for [ROADMAP P2-8 / COMPLIANCE roadmap #4](../ROADMAP.md);
nothing here is implemented.

Read the honesty boundary first: **the tool is not the control.** A.8.24 is an
organizational control the entity owns (a cryptography policy, key management, risk
treatment). qScan/core produce *evidence* that the org *identifies and governs*
its cryptographic usage and is *managing the PQC transition* — they do not satisfy
the control. This is the framing in [COMPLIANCE.md §3](../COMPLIANCE.md); this doc
designs the artifact, not a compliance claim.

## 1. What the control expects (A.8.24)

A.8.24 requires a **defined, enforced policy** on the use of cryptography,
including: which algorithms/parameters are permitted, key management, and
governance of cryptographic mechanisms across their lifecycle. An auditor looks
for evidence that the organization (a) **knows where** cryptography is used,
(b) **assesses** whether it meets policy, and (c) **manages change** — for PQC, the
migration off quantum-vulnerable algorithms.

## 2. The readiness-report artifact

A single, self-describing, **signed and timestamped** document (JSON, with a
human-rendered companion) bundling four parts:

| Part | Source | What it evidences for A.8.24 |
|---|---|---|
| **1. Scan result** | `@qproof/core` `scan()` → `toJson`/`toSarif` | (a) discovery: every candidate classical-crypto usage, with `file:line`, severity, `hndl` flag. |
| **2. Crypto inventory** | `buildInventory()` (`byAlgorithm`/`byCategory`/`bySeverity`, `hndlCount`, `readinessScore`) | (b) assessment: an aggregate posture + a 0–100 readiness score the org tracks over time. |
| **3. CBOM** | CycloneDX cryptographic-asset export (ROADMAP P2-6/COMPLIANCE #1) | a standardized, machine-readable bill of cryptographic assets — the durable inventory record. |
| **4. Attestation** | this report's signed envelope (below) | integrity + provenance: *this* result, from *this* tool version, over *this* repo state, at *this* time. |

### 2.1 Report envelope (sketch)

```jsonc
{
  "reportType": "qproof-readiness",
  "specVersion": 1,
  "subject": {
    "repository": "<git remote/url>",
    "commit": "<full sha>",
    "scannedRoot": "<path>",
    "scanTimeUtc": "2026-06-09T12:00:00Z"
  },
  "tool": { "name": "qScan", "version": "<core VERSION>", "config": { /* effective options */ } },
  "result":    { /* part 1: findings (or a hash + pointer to the SARIF) */ },
  "inventory": { /* part 2: CryptoInventory */ },
  "cbom":      { /* part 3: CycloneDX crypto-assets, or a hash pointer */ },
  "policyMapping": { /* §4: which findings violate which org policy clause, if a policy file is supplied */ },
  "attestation": {
    "contentHash": "sha256:<hash of the canonicalized report body>",
    "timestamp":   { "rfc3161Token": "<base64>", "tsa": "<url>" },
    "signature":   { "alg": "<see §3>", "value": "<base64>", "keyRef": "<key id / cert>" }
  }
}
```

The signature covers the canonicalized body; the timestamp binds it to a point in
time via an **RFC 3161** trusted timestamp (or a transparency-log inclusion proof,
e.g. Sigstore/Rekor) so "point-in-time" is provable, not asserted.

## 3. Signing — honoring the project's own stance

The toolset's whole purpose is to move the world off quantum-vulnerable signatures.
A readiness report that signs itself with classical-only RSA/ECDSA would be a
credibility miss. Design intent:
- Support a **PQC or hybrid** signature for the attestation where the signing
  toolchain allows (ML-DSA / FIPS 204, or a hybrid), so the evidence artifact
  practices the migration it recommends.
- Signing/timestamping uses an **external, vetted** signer — consistent with
  [ADR-0004](../adr/0004-sieve-no-fabricated-vectors.md)'s rule that this project
  performs no cryptography itself; it *orchestrates* a signer, it does not
  implement one.
- Keep the signed body **deterministic** (canonical JSON) so the same scan over the
  same commit produces a verifiable, reproducible hash.

## 4. Mapping findings to policy

If the org supplies a machine-readable cryptography policy (permitted algorithms /
parameter floors — see also the [CONFIG](../CONFIG.md) direction and a future CNSA
2.0 profile), the report's `policyMapping` flags each finding as
`conformant` / `violation` / `transition-pending` against a policy clause. Without
a policy file, the report presents the inventory as neutral evidence and the
**org still owns** the conformance judgment.

## 5. Evidence chain (how an auditor consumes it)

```
scan() ─► inventory ─► CBOM ─┐
                             ├─► canonical report body ─► hash ─► sign + RFC3161 timestamp
org crypto policy (optional)─┘                                         │
                                                                       ▼
              retained, signed, timestamped readiness report  ◄── CI run (audit trail)
```

The audit value comes from **retention + repetition**: the same report emitted on
every CI run, stored, gives a *time series* of readiness scores and a provable
trail that the org continuously identifies and governs its cryptography — exactly
what A.8.24 evidence is.

## 6. Limits (stated up front)

- **Point-in-time, not an audit.** The report attests to *what the scan saw at one
  commit*, signed at one moment. It is **not** an ISO 27001 certification, not an
  auditor's opinion, and not a substitute for the ISMS.
- **Lexical discovery only.** A clean scan is the *absence of detected candidates*,
  not proof of quantum-safety or cryptographic correctness — qScan is
  regex/heuristic ([COMPLIANCE.md](../COMPLIANCE.md) honesty rules).
- **The org owns the control.** Policy, key management, and risk treatment remain
  the organization's; this artifact is one input to A.8.24 evidence, not the
  control itself.
- **Signature ≠ accreditation.** A signed report proves integrity/provenance of the
  *result*; it confers no FIPS/CC/CNSA accreditation.

## 7. Dependencies / sequencing

This artifact depends on, and should follow, two roadmap items: the **CBOM
exporter** (P2-6/COMPLIANCE #1) for part 3, and optionally **CWE tagging** (P2-6)
to enrich part 1. Until those land, parts 1–2 + the signed envelope are still a
viable v0 (CBOM as a hash-pointer placeholder).
